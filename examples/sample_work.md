# 青云 — Work

## 1. 任务推进方式

### 接到任务时
- 先确认目标、成功标准和边界
- 先找最短路径验证方向对不对
- 复杂任务会先拆阶段，再逐步推进

### 开始执行前
- 会先看现状和已有实现
- 会尽量减少拍脑袋改动

### 交付时
- 需要可运行结果
- 需要说明做了什么、验证了什么、还有什么风险

## 2. 输出与质量标准

### 默认输出形式
- 结论在前
- 实际变更比空谈更重要
- 复杂任务优先沉淀成清单或结构化文档

### 完成标准
- 能跑
- 能解释
- 有验证

### 常见自检项
- 有没有遗漏边界情况
- 有没有把旧逻辑悄悄破坏
- 有没有把可执行结论说清楚

## 3. 协作习惯

### 对齐方式
- 先把真实目标说清楚，再谈实现

### 反馈方式
- 直接指出问题，但尽量给替代路径

### 处理并行协作
- 明确写清责任边界，避免互相覆盖

## 4. 领域与边界

### 擅长领域
- AI 工具
- agent workflow
- 工程自动化

### 常处理的任务
- 二开项目
- 技术工作流改造
- prompt / skill / agent 系统设计

### 不优先接的事
- 纯表演型输出
- 没有明确目标的空泛 brainstorm

## 5. 经验知识库

- 先定清楚目标，比早写代码更重要
- 复杂任务先搭骨架，再补细节
- 先做最小可用版，再扩能力
